# 要件定義書：論理思考トレーニング『Deep Why』

**バージョン**: 3.0（エンジニア特化への方向転換）
**最終更新**: 2026-03-17

---

## 1. プロジェクト概要

本システムは、エンジニア向けの論理思考トレーニングツールである。ローカルLLM（Qwen2.5-7B）を使って、エンジニアが日常業務でぶつかったモヤモヤや詰まりを「なぜ？」の問いで言語化し、思考の解像度を上げる。完全ローカル動作により、個人の思考プロセスを外部に漏洩させることなく安全に蓄積・学習する。

### コンセプト

> エンジニアが業務でぶつかったモヤモヤを、「なぜ？」の問いで言語化し、思考の解像度を上げるトレーニングツール。

---

## 2. システムの目的

- **論理的言語化の支援**: 直感や感情を、客観的な論理と具体的な言葉にする力を養う。
- **エンジニアとしての自己理解**: 得意・苦手・強み・価値観を問いを通じて言語化し、自分を知る。
- **業務遂行力の向上**: タスク分解・技術調査・報連相など、日常業務でぶつかる「言語化の壁」を乗り越える力を養う。
- **思考特性の資産化**: 完全ローカル動作の利点を活かし、外部に漏洩させたくない個人の思考プロセスを安全に蓄積・学習する。

---

## 3. ターゲットユーザー

### 主要ターゲット

- 自分の強み・得意なことが感覚ではわかるが言語化できないエンジニア
- 単独で業務を遂行する自信がなく、技術的な判断・タスク分解・報連相に課題を感じているエンジニア
- 成長している実感が持てず、自分がどこにいるのかわからないエンジニア

### ユーザーが抱える悩みの構造

```
単独で業務を行う                説得力をもって説明する
├ 技術的な問題を自己解決できない   ├ 自分の考えに自信がない
├ タスクを分解・見積もりできない   ├ 論理的な話す構成が作れない
└ 調査結果をわかりやすく説明できない └ 適切な報連相・分量感がわからない
          ↓                              ↓
     共通の根っこ：「考えたことを構造化して言葉にする力」がない
          ↓
     Deep Whyが直接解ける問題
```

### 主要な利用シーン

業務でつまずいた瞬間がそのままアプリを開く動機になる。

- 技術調査をしたが結果をうまく説明できない
- タスクをどう分解すればいいかわからない
- レビューでフィードバックをもらったが腑に落ちない
- 自分の強みやキャリアの方向性を言葉にしたい
- 報連相のタイミングや内容に迷っている

## 4. 不便益の設計

本システムは、あえて以下の「不便」を課すことで、ユーザーに「益（知的能力の向上）」を提供する。

| 不便の定義 | 具体的な制約（摩擦） | 得られる益（報酬） |
|:--|:--|:--|
| **認知的負荷** | 抽象的な言葉の使用を禁じ、再定義を求める。 | 思考の解像度の向上。 |
| **回数的制約** | 最低3回・最大5回の対話を終えるまで「結論」を出さない。 | 「思考体力」の養成。 |
| **反復的負荷** | ユーザーの回答が浅い場合、AIが「再考」を促し差し戻す。スコアと採点理由を表示し、差し戻し理由を透明化する。 | 思考の現在地の客観的な可視化と、論理の盲点の発見。 |

## 5. 機能要件

### 5.1 テーマ提案（エンジニア特化）

セッション開始時に、エンジニアの自己理解・業務遂行に関するテーマカテゴリを提示する。ユーザーは提示されたカテゴリから選ぶか、自由入力でテーマを設定できる。

**テーマカテゴリ一覧（初期版）**

| カテゴリ | テーマ例 |
|:--|:--|
| 技術・スキル | 「自分が得意な技術領域はどこか」「今の技術スタックの何が苦手か」 |
| タスク・業務 | 「このタスクをどう分解するか」「なぜ見積もりが外れたか」 |
| コミュニケーション | 「あの報連相はなぜうまくいかなかったか」「説明が伝わらなかった理由」 |
| キャリア・強み | 「自分はどんなエンジニアになりたいか」「自分の強みを一言で言うと何か」 |
| 価値観 | 「どんな仕事にやりがいを感じるか」「なぜエンジニアという職を選んだか」 |
| 自由入力 | ユーザーが任意のテーマを入力 |

### 5.2 思考解像度解析エンジン（Insight Analyzer）

ユーザーの送信後にバックエンドでスコアリングを実行し、結果をUIに表示する。

入力テキストを以下の3つの指標で採点する（0.0〜1.0）。

1. **具体性（Concreteness）**: 5W1H・数値・固有名詞・時期の含有率。「最近」「なんとなく」等の曖昧語は減点。
2. **因果性（Causality）**: 主張に対する根拠の接続が論理的か（飛躍・感情による断定は減点）。
3. **定義度（Definitiveness）**: 独自の意味を持つ言葉（概念語）が客観的に説明されているか。未定義の抽象語の多用は減点。

**スコアのUI表示**：3指標をプログレスバーで表示し、最低指標をハイライト。採点理由を1文で併記する。

### 5.3 適応型「問い」生成（Adaptive Questioning）

採点後、終了条件を満たさない限り毎回問いを生成する。Pythonルールで問いタイプを選択し、Qwen2.5-7Bが問い文を生成する。

| 問いタイプ | 問いタイプの選択基準 | 問いの方向性 |
|:--|:--|:--|
| **垂直深掘り** | 具体性スコアが最低 | いつ・どこで・誰が・どのくらい（例：「なぜそうなったのか？」） |
| **目的回帰** | 定義度スコアが最低 | その言葉・概念の自分なりの定義（例：「その目的の先にある価値は何か？」） |
| **反証提示** | 因果性スコアが最低 | 逆の場合・例外・反例の検討（例：「もしAが成り立たないとしたら、どう説明するか？」） |
| **過去ログ参照** | ChromaDB検索で過去発言との矛盾を検出時 | 過去の発言との整合性 |

### 5.4 終了判定とアウトプット（Completion & Delivery）

- **判定**: 全指標スコアが閾値を超え、かつ往復回数が min_turns（3回）以上の場合に「思考完了」とみなす。max_turns（5回）到達時は強制終了し、現時点の最善を出力する。
- **閾値**: 初期値0.5から始まり、直近5セッションの平均スコアに応じて最大0.85まで引き上げる（成長追従型・自動調整）。
- **出力**: セッション全体を構造化した「論理マップ（Mermaid形式）」および磨き上げられた「主張文」を生成。

### 5.5 エンジニア成長の可視化

セッションの蓄積から、エンジニアとしての自己理解を可視化する。

- **スコア推移グラフ**: セッションを重ねるごとの3指標の変化を折れ線グラフで表示。
- **テーマカテゴリ分布**: どのカテゴリのセッションが多いかを表示し、思考の偏りを把握する。
- **成長ステータス**: 直近の平均スコア・最も伸びた指標・連続セッション日数を表示。

## 6. データ要件（ローカル蓄積・学習）

すべてのデータはローカルPC上に保存し、外部への送信は一切行わない。

### 6.1 SQLite（セッションログ）

以下のデータを保存し、パーソナライズと閾値調整に活用する。

- **思考のデッドエンド**: ユーザーが問いに対して「わからない」と返答したり、同じ言葉を繰り返したり、極端に短い回答しか返せなくなった際のキーワードや文脈。「このテーマになると思考が止まりやすい」というパターンをパーソナライズに活用する。
- **論理の飛躍パターン**: ユーザーが陥りがちな論理的ミス（例：感情による断定）。
- **キーワード／ヒストリー**: ユーザーが特に強い熱量（文字数・回答速度）を示した興味関心領域。
- **スコア履歴**: 各ターンの3指標スコア・問いタイプ・往復回数。
- **閾値履歴**: セッションごとの使用閾値と次回閾値の調整ログ。
- **テーマカテゴリ**: セッションで選択したカテゴリ（技術・タスク・キャリア等）。

### 6.2 ChromaDB（過去コンテキスト参照）

ユーザーの過去発言をベクトル化して蓄積。現在の発言と意味的に近い過去発言を検索し、「過去ログ参照」型の問いの生成トリガーとして使用する。

## 8. 成功の定義

1. ユーザーが業務でぶつかったモヤモヤを、セッションを通じて言語化できること。
2. セッションを重ねるごとに、具体性・因果性・定義度のスコアが統計的に向上すること。
3. 「めんどくさいが、やってよかった」という事後満足度（不便益の実感）の獲得。
4. 自分の強み・苦手・価値観が言葉として蓄積され、エンジニアとしての自己理解が深まること。
